üßÖ Arsitektur Jaringan Tor

Tor (The Onion Router) ‚Äî Jaringan overlay yang menganonimkan trafik internet melalui serangkaian relay terenkripsi. Dikembangkan oleh U.S. Naval Research Laboratory.

Komponen utama:

  • Guard Node ‚Äî Relay pertama, mengetahui IP pengguna
  • Middle Node ‚Äî Relay tengah, tidak mengetahui sumber maupun tujuan
  • Exit Node ‚Äî Relay terakhir, menghubungkan ke internet terbuka
  • Hidden Service ‚Äî Server yang beroperasi di dalam jaringan Tor (.onion)

üîê Onion Routing ‚Äî Enkripsi Berlapis

[Data Asli]
    ‚Üì Enkripsi dengan kunci Exit Node
[Layer 3: Encrypted]
    ‚Üì Enkripsi dengan kunci Middle Node
[Layer 2: Encrypted]
    ‚Üì Enkripsi dengan kunci Guard Node
[Layer 1: Encrypted]
    ‚Üì Kirim ke Guard Node
Node Tahu IP Pengguna? Tahu Tujuan? Tahu Konten?
Guard ‚úÖ Ya ‚ùå Tidak ‚ùå Tidak
Middle ‚ùå Tidak ‚ùå Tidak ‚ùå Tidak
Exit ‚ùå Tidak ‚úÖ Ya ‚ö†Ô∏è Jika non-HTTPS

üßÖ Hidden Services (.onion)

Proses akses hidden service lebih kompleks:

  1. Hidden service membuat introduction points pada beberapa relay
  2. Hidden service mempublikasikan descriptor ke DHT
  3. Klien mendapatkan descriptor dan memilih rendezvous point
  4. Klien mengirim pesan melalui introduction point
  5. Hidden service terhubung ke rendezvous point
  6. Komunikasi berlangsung melalui rendezvous point
Alamat .onion v3: 56 karakter alfanumerik acak dari kunci kriptografi

üéØ Teknik De-anonimisasi Tor

Meskipun Tor dirancang untuk anonimitas, identitas bisa terungkap melalui:

  • Traffic Correlation ‚Äî Menghubungkan trafik masuk & keluar berdasarkan timing
  • Browser Exploitation ‚Äî Eksploitasi kerentanan browser (contoh: NIT oleh FBI)
  • OPSEC Failure ‚Äî Kesalahan operasional pengguna (login akun real)
  • Cryptocurrency Tracing ‚Äî Melacak transaksi Bitcoin
  • Server Misconfiguration ‚Äî Konfigurasi hidden service yang bocor

‚Çø Forensik Cryptocurrency

Cryptocurrency ‚Äî Mata uang digital yang menggunakan kriptografi dan beroperasi secara desentralisasi. Blockchain mencatat semua transaksi secara publik dan permanen.

Komponen utama Bitcoin:

  • Wallet Address ‚Äî Alamat publik (hash dari public key)
  • Private Key ‚Äî Kunci rahasia untuk menandatangani transaksi
  • Transaction ‚Äî Transfer Bitcoin antar alamat
  • Block ‚Äî Kumpulan transaksi yang divalidasi

üîé Pseudonymous ‚â† Anonymous

Bitcoin bersifat pseudonymous ‚Äî semua transaksi tercatat publik di blockchain, tetapi alamat tidak langsung terkait identitas.
Aspek Bitcoin Monero
Ledger Transparan Opaque
Alamat pengirim Terlihat Tersembunyi
Jumlah transaksi Terlihat Tersembunyi (RingCT)
Traceability Dapat dilacak Sangat sulit dilacak

‚õìÔ∏è Teknik Forensik Blockchain

  • Cluster Analysis ‚Äî Mengelompokkan alamat milik entitas yang sama
  • Transaction Graph ‚Äî Memvisualisasi aliran dana antar alamat
  • Exchange Identification ‚Äî Mengenali alamat milik exchange (KYC data)
  • Mixing Detection ‚Äî Mendeteksi penggunaan tumbler/mixer
  • Heuristic Analysis ‚Äî Pola common-input-ownership
Tools: Blockchain.com Explorer, Etherscan.io, Blockchair.com, OXT.me

üìÆ Arsitektur Sistem Email

Komponen Fungsi Contoh
MUA Mail User Agent ‚Äî klien email Outlook, Thunderbird
MSA Mail Submission Agent ‚Äî menerima dari MUA SMTP server

üì° Protokol Email

Protokol Fungsi Port Port TLS
SMTP Pengiriman email 25 465/587
POP3 Unduh email ke lokal 110 995
IMAP Akses email di server 143 993
Implikasi Forensik: POP3 ‚Üí bukti di lokal | IMAP ‚Üí bukti di server + cache lokal

üì¨ Alur Pengiriman Email

alice@kodam.mil.id ‚Üí bob@lantamal.mil.id

1. Alice menulis email di Outlook (MUA)
2. Outlook ‚Üí smtp.kodam.mil.id (MSA/MTA) via SMTP:587
3. MTA kodam ‚Üí DNS MX lookup untuk lantamal.mil.id
4. MTA kodam ‚Üí mx.lantamal.mil.id (MTA penerima) via SMTP:25
5. MTA lantamal ‚Üí MDA ‚Üí Mailbox Bob
6. Bob mengakses email via Outlook/IMAP
üîë Kunci Forensik: Setiap server yang memproses email menambahkan header Received

üìã Struktur Email Header

Field Deskripsi Dapat Dipalsukan?
From Alamat pengirim ‚ö†Ô∏è Ya (spoofable)
Received Jejak routing Sulit dipalsukan
Message-ID ID unik pesan Bergantung server
X-Originating-IP IP pengirim asli Ditambahkan server
Return-Path Alamat bounce Kadang berbeda dari From

üõ°Ô∏è SPF ‚Äî Sender Policy Framework

SPF memungkinkan pemilik domain menentukan server mana yang berhak mengirim email atas nama domain tersebut via DNS TXT record. 
v=spf1 ip4:203.0.113.0/24 include:_spf.google.com -all
Qualifier Simbol Aksi
Pass + Diizinkan (default)
Fail - Ditolak
Softfail ~ Diterima, ditandai
Neutral ? Tidak ada pernyataan

‚úçÔ∏è DKIM ‚Äî DomainKeys Identified Mail

DKIM menggunakan tanda tangan digital untuk memverifikasi bahwa email berasal dari domain yang diklaim dan konten tidak dimodifikasi.

Proses DKIM:

  1. Server pengirim menandatangani header + body dengan private key
  2. Tanda tangan disertakan dalam header DKIM-Signature
  3. Server penerima mengambil public key dari DNS TXT record
  4. Server penerima memverifikasi tanda tangan

üîí DMARC ‚Äî Domain-based Message Authentication

DMARC menggabungkan SPF + DKIM dan memberikan instruksi kepada server penerima jika email gagal autentikasi. 
v=DMARC1; p=reject; rua=mailto:dmarc@kodam.mil.id
Kebijakan Aksi
p=none Monitor saja, tidak ada tindakan
p=quarantine Masukkan ke spam/karantina
p=reject Tolak email sepenuhnya

üîê SPF + DKIM + DMARC = Defense in Depth

Mekanisme Memverifikasi Melindungi Dari
SPF Server pengirim Unauthorized server
DKIM Integritas konten Email tampering
DMARC Kebijakan domain Email spoofing
üí° Forensik: Jika SPF=fail, DKIM=fail, DMARC=fail ‚Üí Indikasi kuat email spoofed!

üî¨ Langkah Investigasi Phishing

  1. Preservasi ‚Äî Simpan email asli (.eml/.msg), jangan forward
  2. Header Analysis ‚Äî Periksa Received, SPF, DKIM, DMARC
  3. URL Analysis ‚Äî Analisis tanpa mengklik, gunakan sandbox
  4. Attachment Analysis ‚Äî Sandbox (Any.run, VirusTotal)
  5. IOC Extraction ‚Äî Kumpulkan IP, domain, hash, email
  6. Threat Intelligence ‚Äî Cek IOC di TI platform
  7. Scope Assessment ‚Äî Berapa banyak penerima terkena?
  8. Documentation ‚Äî Buat laporan forensik lengkap

üïµÔ∏è Deteksi Email Spoofed

üéñÔ∏è Studi Kasus: Phishing di Pangkalan Militer

Skenario: Pangkalan Udara TNI AU menerima laporan email phishing mengaku dari Kemenhan. 3 perwira mengklik tautan dan memasukkan kredensial.

Langkah respons:

  1. Containment ‚Äî Isolasi workstation, block domain phishing
  2. Assessment ‚Äî Identifikasi semua penerima, siapa yang mengklik
  3. Eradication ‚Äî Reset kredensial, scan malware
  4. Recovery ‚Äî Restore dari backup jika perlu
  5. Lessons Learned ‚Äî Update filtering, awareness training
